# CoreWatch 다음 작업 추가 항목 — GitHub 브랜치 보호

이 문서는 다음 작업에서 GitHub 저장소 `Wakbu/CoreWatch`의 중요 브랜치 보호 상태를 점검하고 룰셋을 적용하기 위한 작업 항목입니다. 오늘은 저장소 설정을 변경하지 않습니다.

## 현재 경고의 의미

GitHub의 `Your main branch isn't protected` 경고는 기본 브랜치인 `main`에 강제 푸시, 삭제 또는 검증되지 않은 변경의 직접 반영을 제한하는 보호 규칙이 없다는 뜻입니다.

## 다음 작업에서 먼저 확인할 내용

- 저장소 기본 브랜치와 실제 배포 기준 브랜치가 `main`인지 확인합니다.
- 현재 GitHub Actions 워크플로와 필수 상태 검사 이름을 확인합니다.
- 저장소 소유자 계정과 현재 인증 토큰에 룰셋 관리 권한이 있는지 확인합니다.
- 혼자 개발하는 저장소인지, 다른 협업자의 리뷰가 필요한 저장소인지 확인합니다.
- 기존 작업 브랜치와 Draft PR이 새 규칙에 의해 막히지 않는지 확인합니다.

## 권장 `main` 브랜치 룰셋

- 브랜치 삭제 금지
- 강제 푸시 금지
- 변경 사항은 Pull Request를 통해서만 병합
- 필수 빌드 및 무결성 검사 통과 후 병합
- 해결되지 않은 PR 대화가 있으면 병합 금지
- 관리자 우회는 비상 복구 목적에 한해 유지하거나 엄격히 제한

다음 항목은 저장소 운영 방식 확인 후 선택합니다.

- 필수 승인 1회: 협업자가 있을 때 권장하며, 단독 개발이면 자기 PR을 스스로 승인할 수 없어 작업을 막을 수 있습니다.
- 최신 `main` 기준 상태 검사 요구: 안전성은 높지만 매 변경 시 재검사가 필요합니다.
- 선형 커밋 기록 요구: Squash 또는 Rebase 병합만 사용할 경우 적용합니다.
- 서명된 커밋 요구: 현재 커밋 서명 환경을 먼저 구성한 뒤 적용합니다.

## 적용 방법

권한이 허용되면 GitHub Rulesets API 또는 저장소 설정 화면에서 적용합니다. 적용 전 현재 설정을 조회해 기록하고, 적용 후 다시 조회해 실제 활성 상태를 검증합니다.

## 완료 조건

- `main` 브랜치의 강제 푸시와 삭제가 차단되어야 합니다.
- Pull Request 없이 `main`에 직접 변경을 반영할 수 없어야 합니다.
- CoreWatch의 빌드 및 무결성 검사에 대응하는 필수 상태 검사가 설정되어야 합니다.
- 정상적인 작업 브랜치 푸시와 PR 생성은 계속 가능해야 합니다.
- 적용된 룰셋 구성과 복구 방법을 프로젝트 문서에 기록해야 합니다.
